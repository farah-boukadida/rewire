# 场景：HR 技术岗招聘，24 小时简历筛选 + 自动打招呼

> **用户描述**：我是 HR，负责技术岗招聘。每天要在 Boss 直聘上翻大量简历，对比我们的 JD，找到合适的候选人然后打招呼。一个岗位可能要翻几百份简历才能找到几个合适的，每天花 3-4 小时在上面，眼睛都看花了。而且好的候选人很抢手，我第二天才看到的简历，别的公司可能已经联系过了。我希望有个 Agent 能 24 小时帮我盯着 Boss 直聘，有合适的简历自动比对 JD 推荐给我，最好能自动打招呼。

---

## 你现在的问题

> 你不是在招聘，你是在当一台人肉筛选机——几百份简历过你的眼睛，99% 被丢掉，1% 被留下，而你的大脑在做的事情和 `if (简历.技能 match JD.要求) then 打招呼` 没有本质区别。更要命的是，你是一台只有工作时间才开机的筛选机。好候选人的简历凌晨 2 点刷新了，你第二天 9 点才看到，7 个小时里已经有 3 家公司给他打了招呼。你输的不是眼力，是时差。

---

## 闭环拆解

### 闭环 A：简历筛选（核心吞吐量瓶颈）

| 步骤 | 现在谁在做 | 具体动作 | 状态 |
|------|-----------|---------|------|
| **调研** | 人肉 | 打开 Boss 直聘网页版，按岗位搜索候选人，逐页翻简历列表 | 意愿瓶颈（最机械、最累眼） |
| **决策** | 人肉 | 看每份简历的工作经验、技能、学历，对比 JD 判断是否合适 | 吞吐量瓶颈（耗时最多），80% 可被规则覆盖 |
| **规划** | 人肉 | 想打招呼话术，根据候选人背景微调 | 有判断价值，但模板化程度高 |
| **执行** | 人肉 | 逐个点进简历 → 打招呼 → 发消息 | 纯重复点击 |
| **反馈** | 被动 | 候选人回不回全靠运气，没有数据跟踪 | 不知道哪种话术有效 |
| **复盘** | 缺失 | 从没统计过筛选命中率、回复率、各渠道转化 | 凭感觉招人 |

### 闭环 B：候选人跟进

| 步骤 | 现在谁在做 | 具体动作 | 状态 |
|------|-----------|---------|------|
| **调研** | 人肉 | 回到 Boss 直聘看谁回了消息 | 分散在一天中反复检查 |
| **决策** | 人肉 | 判断候选人的回复意向：强意向/一般/拒绝 | 有判断价值 |
| **规划** | 人肉 | 安排面试时间、准备候选人资料给面试官 | 有判断价值 |
| **执行** | 人肉 | 拉群、发面试邀请、同步信息 | 部分可自动化 |
| **反馈** | 被动 | 面试官口头反馈 | 不结构化 |
| **复盘** | 缺失 | 没有漏斗数据 | 不知道卡在哪一步 |

两条闭环共 12 步，10 步人肉，2 步缺失。你的时间被"翻简历"这一步吞掉了 80%，真正需要你判断力的"面试安排"和"候选人沟通"反而没时间做好。

---

## Claude Code 怎么驱动这一切

**机制**：Claude Code 运行在你的电脑（或一台常开的 Mac mini）上，通过 bb-browser 直接操作你浏览器里的 Boss 直聘网页版——用的就是你已登录的账号，不需要破解 API、不需要额外账号。它每隔 15-30 分钟自动醒来，打开 Boss 直聘的"推荐牛人"和搜索页面，逐个读取候选人信息，用你在 CLAUDE.md 里定好的 JD 匹配规则打分，高分的自动打招呼，中等的推给你确认，不匹配的跳过。配合 `launchd` 定时任务，它 24 小时不间断工作。

**故事**：

凌晨 1 点，一个有 5 年 Go 后端经验的候选人刷新了简历。

Claude Code 15 分钟后自动醒来，用 bb-browser 打开 Boss 直聘的推荐牛人页面。它 `snapshot` 拿到页面结构，看到候选人列表。逐个点进去，读取每份简历的关键信息——工作年限、技术栈、学历、最近工作经历。

这个候选人：5 年经验、Go + Kubernetes + 微服务、985 本科、前两段经历都在中大厂。Claude Code 拿这些信息对比你定好的 JD——你要的是"3 年以上 Go 开发，熟悉云原生，本科及以上"。完美匹配，打分 92/100。

它按你预设的话术模板，自动组装一条打招呼消息：

> "你好！看到你在云原生和 Go 微服务方向有丰富经验，我们团队正在做 XX 方向的技术架构升级，目前在找高级后端工程师，base 北京，薪资 40-60K，感兴趣的话可以聊聊？"

bb-browser 点击"打招呼"按钮，填入消息，发送。同时在本地 `data/candidates.jsonl` 里记一条：

```json
{"name": "张XX", "score": 92, "skills": ["Go","K8s","微服务"], "action": "auto_greet", "time": "2026-03-06T01:15:00"}
```

然后它给你发一条 Bark 通知，level 设成 `passive`，不吵你睡觉：

> "已自动打招呼：张XX，5 年 Go 后端，匹配度 92 分"

凌晨 3 点，又来一个候选人，3 年经验、Java + Spring Boot，但简历里提到"正在学习 Go，有个人项目用 Go 写的"。Claude Code 打分 65——不完全匹配，但有潜力。它不自动打招呼，而是存到待确认列表，Bark 推一条 `active` 级别的通知：

> "待确认：李XX，3 年 Java 转 Go 方向，匹配度 65 分，是否打招呼？"

你早上 8 点起床，手机上整整齐齐列着一晚上的结果：

- "夜间巡检摘要：浏览 47 份简历，自动打招呼 3 人（92/88/85 分），待确认 2 人（65/61 分），跳过 42 人"

你点开待确认的两份，看了 30 秒——那个 Java 转 Go 的背景不错，你回一句"打招呼"。Claude Code 立刻执行。另一个学历不符，你回"跳过"。

9 点到公司坐下，你打开 Boss 直聘，发现昨晚自动打招呼的那个 92 分候选人已经回了消息："可以聊聊，方便加微信吗？"——因为你是凌晨 1 点就联系了他，比其他公司早了 8 个小时。

这个候选人，如果等你今天 9 点才看到简历，他的消息列表里可能已经排了 5 家公司。但你是第一个联系他的。

---

## 你的工作会变成什么样

早上 8 点，你还在刷牙，手机 Bark 通知已经汇总好了：

- "过夜筛选报告：浏览 130 份简历，自动打招呼 5 人（均 85 分以上），待确认 4 人，已收到 2 条回复"

你到公司第一件事不是打开 Boss 直聘翻简历，而是看候选人的回复，安排面试。翻简历这件事，再也不用你干了。

你的工作重心从"大海捞针"变成了"看鱼上钩"：

- 每天花 10 分钟确认几个待定候选人
- 花 20 分钟处理回复、安排面试
- 剩下的时间，你可以去做真正需要 HR 判断力的事：跟候选人深聊、优化 JD、跟面试官对齐标准、分析招聘漏斗数据

3-4 小时 → 30 分钟。而且 24 小时不间断，再也不怕好候选人被别人抢先联系。

## 怎么搭建

### 1. 准备工具

```bash
# bb-browser（浏览器自动化，操作 Boss 直聘网页版）
bb-browser status  # 确认 daemon 在跑

# jq（处理 JSON 数据）
brew install jq
```

Boss 直聘没有公开 API，也没有成熟的 CLI 工具。现有的开源方案（如 GitHub 上的 boos-helper，1.2k stars，Chrome 扩展）和 RPA 平台（影刀 RPA、八爪鱼 RPA）都是走浏览器自动化路线。我们用 bb-browser 做同样的事，但由 Claude Code 来驱动决策，比纯规则 RPA 更灵活——它能真正"读懂"简历内容，而不是只做关键词匹配。

### 2. 前置准备：Boss 直聘账号

- 确保你有 Boss 直聘**招聘者账号**（企业版）
- 在 Chrome 里登录好，bb-browser 会复用这个登录态
- 建议开通付费功能（扩大每日打招呼额度，牛人推荐更精准）

### 3. 创建项目目录

```bash
mkdir ~/hr-recruit-agent && cd ~/hr-recruit-agent
mkdir -p data logs
```

### 4. 编写 CLAUDE.md

```markdown
# HR 招聘 Agent —— Boss 直聘 24 小时简历筛选

## 身份
你是技术岗招聘助手，负责 24 小时监控 Boss 直聘上的候选人简历，自动筛选并打招呼。

## 工具
- bb-browser：操作 Boss 直聘网页版（复用浏览器登录态）
- Bark：给 HR 发手机通知
- 本地文件系统：记录候选人数据和操作日志

## 在招岗位

### 高级 Go 后端工程师
- 搜索关键词：Go 后端工程师
- 必须满足（硬性）：
  - 工作经验 >= 3 年
  - 技术栈包含 Go（Golang）
  - 学历：本科及以上
- 加分项（软性）：
  - 有 Kubernetes / 云原生经验 +15 分
  - 有微服务架构经验 +10 分
  - 985/211 学历 +10 分
  - 有大厂经验（字节/阿里/腾讯/美团等）+10 分
  - 有开源项目贡献 +10 分
- 基础分：满足所有硬性条件 = 60 分，加上软性加分
- 自动打招呼阈值：>= 80 分
- 待确认阈值：60-79 分
- 跳过：< 60 分

### [其他岗位按同样格式添加]

## 打招呼话术模板

### 自动打招呼（>= 80 分）
"你好！看到你在{候选人核心技能方向}有丰富经验，我们是{公司名}，正在做{业务方向}，目前在找{岗位名}，base {城市}，薪资{薪资范围}。你的背景很匹配，方便聊聊吗？"

### 待确认（HR 审核后发送）
先存草稿，等 HR 确认后再发。

## Boss 直聘操作流程

### 推荐牛人巡检
1. bb-browser open https://www.zhipin.com/web/boss/recommend
2. snapshot 获取推荐列表
3. 逐个点进候选人详情页
4. 读取：姓名、工作年限、学历、技术栈、最近工作经历、期望薪资
5. 按评分规则打分
6. 按阈值决定操作（自动打招呼 / 存待确认 / 跳过）

### 搜索牛人巡检
1. bb-browser open https://www.zhipin.com/web/boss/search/geek
2. 按岗位搜索关键词搜索
3. 翻阅前 5 页结果（每次巡检）
4. 对每个候选人执行同上流程

### 打招呼操作
1. 在候选人详情页找到"打招呼"按钮
2. snapshot -i 获取按钮 ref
3. click 打招呼按钮
4. 在弹出的消息框里 fill 话术内容
5. 点击发送
6. 记录到 data/candidates.jsonl

## 消息回复巡检
1. bb-browser open https://www.zhipin.com/web/boss/chat
2. 检查是否有新消息
3. 有回复的候选人 → Bark 通知 HR（level: timeSensitive）
4. 不自动回复聊天消息——聊天环节必须 HR 亲自介入

## 数据记录
每次操作记录到 data/candidates.jsonl：
- 候选人基本信息（脱敏）
- 匹配分数和各项得分明细
- 操作类型（auto_greet / pending / skip）
- 时间戳

每日汇总记录到 data/daily-{date}.json：
- 浏览简历总数
- 各分数段分布
- 自动打招呼数
- 待确认数
- 收到回复数
- 回复率

## 通知规则
- 自动打招呼后 → Bark 推送，level: passive
- 待确认候选人 → Bark 推送，level: active
- 候选人回复消息 → Bark 推送，level: timeSensitive
- 系统异常（登录失效、页面结构变化、打招呼次数到上限）→ Bark 推送，level: critical

## 安全规则
- 每次巡检间隔不少于 15 分钟，避免触发平台风控
- 操作之间加 3-5 秒随机延迟，模拟人类操作节奏
- 每日打招呼上限：跟随平台规则，不超过平台限额
- 如果检测到验证码或异常拦截，立即停止并通知 HR
- 聊天消息只读不写——自动回复候选人消息风险太高，必须人工处理
```

### 7. 配置 24 小时定时运行

验证交互模式没问题后，配置 `launchd` 让它每 30 分钟自动跑一次：

```bash
# 创建巡检脚本
cat > ~/hr-recruit-agent/patrol.sh << 'EOF'
#!/bin/bash
cd ~/hr-recruit-agent
claude --print --dangerously-skip-permissions \
  -p "执行定时巡检：1) 打开 Boss 直聘推荐牛人页面 2) 筛选候选人并按规则打分 3) 80分以上自动打招呼 4) 60-79分存待确认通知我 5) 检查是否有新回复 6) 记录数据到 data/" \
  2>&1 | tee -a logs/patrol-$(date +%Y%m%d-%H%M).log
EOF
chmod +x ~/hr-recruit-agent/patrol.sh

# 创建日报脚本
cat > ~/hr-recruit-agent/daily-report.sh << 'EOF'
#!/bin/bash
cd ~/hr-recruit-agent
claude --print --dangerously-skip-permissions \
  -p "生成今日招聘日报：汇总 data/ 里今天的巡检记录，统计浏览/打招呼/回复数据，生成漏斗报告，通过 Bark 发送汇总" \
  2>&1 | tee -a logs/report-$(date +%Y%m%d).log
EOF
chmod +x ~/hr-recruit-agent/daily-report.sh
```

macOS `launchd` 定时任务配置：

```xml
<!-- ~/Library/LaunchAgents/com.hr.recruit.patrol.plist -->
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>Label</key>
    <string>com.hr.recruit.patrol</string>
    <key>ProgramArguments</key>

        <string>/bin/bash</string>

</plist>
```

```bash
# 日报每天晚上 9 点生成

# 添加：
# 0 21 * * * /Users/你的用户名/hr-recruit-agent/daily-report.sh
```

## 运行模式：三档混合

| 模式 | 触发方式 | 覆盖场景 | 你的介入 |
|------|---------|---------|---------|
| **24 小时无人值守** | launchd 每 30 分钟 | 简历筛选 + 高分自动打招呼 | 看 Bark 通知即可 |
| **定时模式** | cron 每天 1 次 | 日报生成 + 漏斗统计 | 不介入 |
| **交互模式** | 你主动启动 | 审核待确认候选人、处理回复、调整 JD 和评分规则 | 你坐在电脑前对话 |

关键：无人值守处理 80% 的筛选和打招呼动作，你只处理"中间地带"的候选人和实际的对话沟通。

---

## 风险与边界

**平台风控**：Boss 直聘有反自动化检测。方案里已经设计了操作间隔（3-5 秒随机延迟）、巡检频率限制（30 分钟一次）、每日打招呼上限跟随平台规则。但仍有被风控的可能。建议：先用低频率跑一周，观察账号状态，没问题再逐步提频。如果出现验证码，Agent 会立即停止并通知你。

**聊天不自动回复**：打招呼是标准化的第一句话，风险可控。但候选人回复后的对话涉及薪资谈判、岗位细节、面试安排，必须你亲自处理。Agent 只负责"发现候选人 + 打第一声招呼"，后续对话交给你。

**登录态维护**：Boss 直聘的登录态可能会过期。Agent 如果检测到被重定向到登录页，会立即发 Bark 通知（level: critical）让你重新登录。

---

## 元层：它会越来越好

**评分规则校准**：每周看一次"打了招呼但没回复"和"打了招呼且回复积极"的候选人对比。如果发现某个加分项（比如"大厂经验"）对回复率没有正相关，降低权重；如果发现某个你忽略的特征（比如"有技术博客"）和高回复率正相关，加上去。Agent 帮你统计数据，你调规则，一行 CLAUDE.md 的事。

**话术 A/B 测试**：准备 2-3 版打招呼话术，Agent 按比例随机使用，一周后统计各版本回复率。效果差的淘汰，效果好的留下。你不用猜"哪种话术好"，数据会告诉你。

**岗位画像学习**：Agent 积累一个月数据后，会沉淀出每个岗位"最终入职的候选人"长什么样——不是你写在 JD 里的理想画像，而是实际通过面试并入职的人的真实画像。这个画像可能会让你发现：你一直在要求"5 年经验"，但实际入职的人平均只有 3.5 年；你从没要求"有开源贡献"，但入职的人里 60% 都有。这些洞察反过来帮你优化 JD 本身。

**招聘漏斗量化**：浏览 → 打招呼 → 回复 → 面试 → 入职，每一步的转化率第一次被完整记录下来。你终于可以跟老板用数据说话："上周浏览了 900 份简历，打招呼 35 人，回复 12 人，面试 4 人，通过 1 人。瓶颈在面试通过率，不是简历量——我们需要调整的是面试标准，不是让我翻更多简历。"
