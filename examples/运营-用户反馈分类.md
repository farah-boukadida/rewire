# 场景：运营每天手动分类用户反馈

> **用户描述**：我是运营，每天从后台系统手动导出用户反馈 Excel，逐条阅读判断是 Bug、需求还是咨询，整理成表格发到飞书群给研发。每天 50-80 条，花 2-3 小时。

---

## 你现在的问题

> 你每天花 2-3 小时做的事，本质上是人肉充当分类 API——上游系统没有接口，下游研发没有自助入口，所有信息必须经过你的眼睛和手才能流动。你不是在做运营判断，你是在当系统和系统之间的人肉管道。

---

## 闭环拆解

| 步骤 | 现在谁在做 | 具体动作 | 状态 |
|------|-----------|---------|------|
| **调研** | 人肉 | 登录后台 → 翻页 → 点导出 → 等下载 → 打开 Excel | 意愿瓶颈（最烦的） |
| **决策** | 人肉（可规则化） | 逐条读反馈内容，凭经验判断 Bug/需求/咨询 | 吞吐量瓶颈（耗时最多），90% 可被规则覆盖 |
| **规划** | 人肉 | 按分类结果重新整理成分组表格 | 纯格式搬运 |
| **执行** | 人肉 | 复制表格贴到飞书群 | 纯粘贴操作 |
| **反馈** | 被动/缺失 | 研发偶尔说"分类不对" | 没有结构化回收 |
| **复盘** | 缺失 | 分类规则从未被显性化、从未被迭代 | 规则锁死在第一版 |

六步里四步纯人肉，两步缺失。整条链路没有闭环。

## Claude Code 怎么驱动这一切

**机制**：Claude Code 是一个运行在你电脑上的 Agent。它读取项目目录里的指令文件（CLAUDE.md），知道自己该做什么、有什么工具可用。然后它像一个坐在你电脑前的同事一样，用 shell 命令调用各种工具来完成工作——用 bb-browser 操作你的浏览器登后台导出数据，用 Python 脚本处理 Excel，用 curl 调飞书 Webhook 发消息。它能思考、能判断、能执行，每一步都在你的电脑上真实发生。

**故事**：

你早上到工位，打开终端，跟 Claude Code 说一句："处理今天的反馈"。

它先用 bb-browser 打开你的后台系统——用的就是你浏览器里的登录态，不需要额外输密码。它 `snapshot -i` 拿到页面结构，找到"导出"按钮，点击，等 Excel 下载完成。这一步你最烦的 5 分钟，它 30 秒搞定。

然后它用 Python 读取 Excel，逐条分析反馈内容。它按你定好的规则来判断——提到"崩溃""报错""闪退""打不开"的归 Bug，提到"希望""建议""能不能""要是能"的归需求，问"怎么用""如何""在哪里"的归咨询。90% 的条目它直接分好类。剩下拿不准的，它会问你：

> "第 23 条'支付完成后页面一直转圈'——这算 Bug 还是咨询？"

你回一句"Bug"，它记住，继续。5-8 条不确定的，你 3 分钟处理完。

分完之后，它把结果按 Bug / 需求 / 咨询分组，组装成飞书消息卡片，用 `curl` 调你群里机器人的 Webhook 直接发出去。研发打开飞书就能看到格式清晰的分类表格。

全程你就像在跟一个懂行的助手对话——它干活，你只在关键节点拍板。

---

## 你的工作会变成什么样

你早上到工位，打开飞书，反馈已经分好类发在群里了。你的待办里只有 5-8 条标了"不确定"的，点开扫一眼，改两个分类，确认三个没问题，3 分钟搞定。然后你回一句"发出去"，消息就到群里了。

2-3 小时 → 5 分钟。

剩下的时间，你可以去看看最近反馈里有没有反复出现的高频问题——比如上周有 12 条都提到"支付超时"，这才是运营该做的事：发现趋势、推动产品改进，而不是当分拣机器。

## 怎么搭建

### 1. 准备工具

```bash
# bb-browser（浏览器自动化，确保 Chrome 打开且 daemon 在跑）
bb-browser status

# Python 依赖（处理 Excel）
pip install openpyxl
```

### 2. 创建项目目录

```bash
mkdir ~/feedback-agent && cd ~/feedback-agent
```

### 3. 编写 CLAUDE.md

```markdown
# 用户反馈处理 Agent

## 任务
每天从后台导出用户反馈 Excel，分类后发到飞书群。

## 后台系统
- URL：<你的后台地址>
- 导出路径：<登录后点哪些菜单到导出页面>
- 使用 bb-browser 操作，复用浏览器登录态

## 分类规则
- Bug：崩溃/闪退/报错/打不开/白屏/卡死/无响应/加载失败
- 需求：希望/建议/能不能/要是能/最好能/应该/可以加
- 咨询：怎么/如何/在哪/为什么/是不是（疑问语气）
- 不确定：以上都不明显匹配的，标记为不确定，等人工确认

## 飞书 Webhook
- 地址：<你的 Webhook URL>
- 消息格式：按 Bug/需求/咨询 分组，每组列出条目编号和内容摘要

## 工作流程
1. bb-browser 打开后台 → 导航到导出页 → 点击导出 → 等待下载
2. Python 读取 Excel → 逐条分类 → 不确定的问我
3. 组装飞书消息 → curl 发送到 Webhook
```

### 4. 需要你提供的信息

- 后台地址和导出页面的操作路径
- Excel 的列结构（反馈内容在第几列）
- 飞书 Webhook 地址

### 5. 启动验证

```bash
cd ~/feedback-agent
claude
# 说："处理今天的反馈"
# 第一次全程盯着，确认每步对不对
```

## 元层：它会越来越好

**分类规则迭代**：每次你纠正一个分类，把判断依据告诉它。它建议你更新 CLAUDE.md 里的规则，下次同类表述就能自动处理。不确定比例会从 10% 逐步降到 2-3%。

**质量回检**：每周统计分类准确率、不确定比例、被研发纠正的条数。偏差大了，补规则。

**格式优化**：研发反馈"能不能加优先级"或"Bug 按模块分"，直接改 CLAUDE.md 一行文字，下次就生效。
